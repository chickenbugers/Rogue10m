# Rogue10m 프로젝트 개발 기준

## 브랜치 운영

- 기능 개발은 항상 `feature/*`, `content/*`, `level/*`, `balance/*`, `fix/*` 계열 브랜치로 이동한 뒤 진행한다.
- 통합 흐름은 `develop` -> `test` -> `main` 순서를 기본으로 한다.
- 커밋 전에는 변경 파일과 변경 목적을 먼저 요약하고, 사용자의 명시적인 확인을 받은 뒤 커밋한다.

## 기록 규칙

- 코드, 설정, 문서, 브랜치, 빌드, 프로젝트 구조가 바뀌면 `DevLog/YYYYMMDD.txt`에 한국어로 기록한다.
- DevLog는 기존 내용을 덮어쓰지 않고 이어서 작성한다.
- 기능 개발 결과는 Tistory에 옮기기 쉬운 형식으로 정리하고, Notion에 옮길 후보 요약을 별도로 표시한다.

## 캐릭터 기능 구조

- 장기적으로 유지되어야 하는 플레이어 상태는 `PlayerState`를 우선 활용한다.
- 캐릭터 클래스가 커질 경우 인벤토리, 전투, 숙련도, 재화, 성장, 상태 이상 같은 기능은 `ActorComponent`로 분리한다.
- 현재 프로토타입 단계에서는 동작 검증을 우선하되, 기능이 안정되면 데이터와 로직의 책임을 분리한다.

## 데이터 설계

- 무기, 아이템, 스킬, 몬스터, NPC, 밸런스 수치 같은 확장 데이터는 `Data Asset` 기반으로 전환하는 것을 목표로 한다.
- C++ 클래스는 동작과 검증 규칙을 담당하고, 에디터에서 조정할 값은 `UPROPERTY`로 노출한다.
- Blueprint는 에셋 연결과 시각적 조정 역할을 맡고, 핵심 게임플레이 로직은 C++에 둔다.
